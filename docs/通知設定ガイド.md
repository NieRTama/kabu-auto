# 通知設定ガイド（Discord Webhook）

kabu-auto は運用中の重要イベントを Discord 経由でスマートフォン等へ通知できる。
本書は設定方法・通知トリガー・実装の挙動・将来の通知プロバイダ追加方法をまとめたもの。

> **旧LINE Notifyについて**: 以前は LINE Notify を使用していたが、提供元（LINE社）に
> より2025年3月31日にサービス終了し、復活の見込みがないため本プロジェクトでは
> 完全に廃止し、Discord Webhook へ置き換えた。

---

## 1. 何のための機能か

`alert(title, message)`（[src/core/alerts.py](../src/core/alerts.py)）が呼ばれるたびに:
1. 必ず `logger.warning` でログへ記録する（通知設定の有無に関わらず常に実行）。
2. 設定済みの通知プロバイダ（現状はDiscord Webhookのみ）へ同じ内容を追加送信する。

つまり通知は「ログを見ていなくても気づけるようにする」ための**補助的な通知手段**であり、
必須の監視経路ではない（未設定でも動作上の問題はなく、単に外部送信だけがスキップされる）。
停止に気づく主手段は引き続きダッシュボード・ログファイル・`/api/halt` 等の確認とする。

## 2. Discord Webhook URL の取得方法

1. 通知を受け取りたいDiscordサーバーのテキストチャンネルを開く。
2. チャンネル設定（歯車アイコン）→「連携サービス」→「ウェブフック」→「新しいウェブフック」。
3. 任意の名前・アイコンを設定し、「ウェブフックURLをコピー」を押す。

このURL自体が秘密情報（認証ヘッダー不要でこのURLにPOSTするだけで送信できる）。
他者に共有しないこと。

## 3. プロジェクトでの設定方法

**推奨: `.env` に設定する**（`config.yaml` への平文保存を避けるため）。

```bash
# .env.example をコピーして使う
cp .env.example .env
```

`.env` に以下を追記:

```
DISCORD_WEBHOOK_URL=<コピーしたWebhook URL>
```

[src/core/config.py](../src/core/config.py) の `_apply_env_overrides()` が起動時に
`DISCORD_WEBHOOK_URL` 環境変数を読み、`config.yaml` の `alerts.discord_webhook_url` を
上書きする（環境変数が優先）。`config.yaml` に直接書く方法もあるが、リポジトリへの
誤コミットで漏洩するリスクがあるため非推奨:

```yaml
# config.yaml（非推奨。.envを使うこと）
alerts:
  discord_webhook_url: ""
```

設定を反映するには kabu-auto の再起動が必要（起動時に一度だけ読み込む）。

## 4. 通知が送られるタイミング

`alert()` は以下のイベントで呼ばれる（[src/core/alerts.py](../src/core/alerts.py) 経由）。
カテゴリごとに整理する。

### リスク・損益関連
- **損切り実行**: 損切りラインに到達し成行売りを発注したとき（[src/services/trading.py](../src/services/trading.py)）
- **異常検知**（[src/core/health.py](../src/core/health.py) の定期ジョブ、新規発生時のみ通知）:
  - 未解決注文（UNKNOWN/CANCEL_FAILED）の残存
  - 当日損失が上限に接近/到達
  - 当日合計ドローダウン（実現損失＋含み損）が上限に接近/到達
  - 取引停止スイッチ（kill switch）がON
  - 終値取得不可で含み損益に未反映の銘柄がある

### 注文・約定関連（[src/execution/order_manager.py](../src/execution/order_manager.py)）
- 起動時同期でAPIに見つからない注文を UNKNOWN にしたとき
- 建玉ドリフト検知（DBとブローカー実建玉の不一致）
- 注文照合中にブローカー側で注文が見つからなくなったとき
- 損切り/緊急決済前の競合注文キャンセルに失敗し、退出注文をブロックしたとき
- 緊急全決済が `/positions` 取得失敗でブロックされたとき
- 取引停止スイッチ（kill switch）作動時
- semi_live承認が`APPROVING`のまま中断されて起動時に検知されたとき
- semi_live: 新規の発注承認待ちが発生したとき
- semi_live: 承認後の発注は成立したが記録更新に失敗したとき（要確認）
- 注文キャンセルの失敗・拒否

### MLモデル関連（[src/strategy/ml_model.py](../src/strategy/ml_model.py)）
- モデルファイルのSHA256が記録と不一致（改ざん/破損の疑い）
- モデルのメタ情報（ハッシュ記録）が無く、ロードを拒否したとき

## 5. 実装の挙動・制約

- **失敗は飲み込む**: Discord送信が失敗（タイムアウト・URL失効・APIエラー等）してもアプリは
  継続し、`logger.error` にのみ記録する。送信失敗時の再送・リトライは行わない。
- **タイムアウト10秒**: `requests.post` に `timeout=10` を指定済み。
- **Webhook URLはログに出さない**: 送信内容（タイトル・本文）はログに出るが、URL自体は
  出力しない（失敗時のログにはプロバイダ名のみ記録）。
- **メッセージ形式**: `【kabu-auto】{title}\n{message}` の形式で1通ずつ送信する（バッチ化なし）。
- **本文の長さ制限**: Discordの本文上限は2000文字。超過時は自動的に末尾を切り詰め、
  省略マーク `…(省略)` を付けて送信する（送信エラーで通知が完全に消えるより、要点が欠けても
  通知自体が届く方を優先）。
- **頻度制限なし**: アプリ側でのレート制限は行っていない（短時間に多数のイベントが
  発生すると、その分だけDiscord通知が連続する）。
- **複数プロバイダ対応**: 設定済みの全プロバイダへ送信を試み、1つが失敗しても他のプロバイダへの
  送信は継続する（現状はDiscordのみのため実質1件だが、§6の手順で追加した場合も同様）。

## 6. 動作確認方法

設定後、軽量に動作確認したい場合は Python REPL から直接呼び出せる:

```python
import src.core.config as cfg
cfg.load("config.yaml")
from src.core.alerts import alert
alert("動作確認", "Discord通知連携テストです")
```

Discordに通知が届けば設定は正しい。届かない場合は、まずログに `通知送信失敗` が
出ていないか確認する（URLの誤り・Webhookの削除等が主な原因）。

## 7. 将来の通知プロバイダ追加方法

Discord以外（Slack/Telegram/email等）を追加したい場合、`alert()` および17箇所の
呼び出し側を変更する必要はない。以下の4ステップのみでよい。

1. `config.yaml` の `alerts:` セクションに新キー（例: `slack_webhook_url: ""`）を追加。
2. `.env.example` に対応する環境変数（例: `SLACK_WEBHOOK_URL=`）を追加し、
   [src/core/config.py](../src/core/config.py) の `_apply_env_overrides()` に
   上書き処理を1行追加。
3. [src/core/alerts.py](../src/core/alerts.py) に新しいプロバイダクラス
   （`AlertProvider` の `send(message: str) -> None` を満たすクラス）を実装。
4. `build_providers()` に「設定されていれば追加する」分岐を1つ追加。
