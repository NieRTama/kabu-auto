### 1.1 src/core/config.py — 設定管理

**責務**: `config.yaml` の読み込みとアプリ全体への提供

| 関数 | 引数 | 戻り値 | 説明 |
|------|------|--------|------|
| `load(path)` | path: str | dict | YAMLを読み込みキャッシュ。読み込み後に `_apply_env_overrides()` を実行 |
| `get()` | なし | dict | キャッシュ済み設定を返す |
| `get_section(section)` | section: str | dict | セクション単位で設定を返す |
| `_apply_env_overrides(config)` | config: dict | なし | 環境変数を `config.yaml` の値に優先して適用（下記） |

`_apply_env_overrides()` が上書きする環境変数:

| 環境変数 | 上書き先 |
|----------|----------|
| `KABU_API_PASSWORD` | `kabu_station.password` |
| `KABU_DASHBOARD_TOKEN` | `dashboard.api_token` |
| `LINE_NOTIFY_TOKEN` | `alerts.line_notify_token` |

> **機密情報の取り扱い**: `config.yaml` はgit管理されるため、パスワードやトークンを
> 平文で書かない運用とする。`main.py` 起動時に `python-dotenv` で `.env`（`.gitignore`済み）を
> 読み込み、上記の環境変数を `config.yaml` の値に優先して適用する。
> `config.yaml` には `kabu_station.password` 項目を**置かない**（既定で削除済み）。`load()` は
> YAMLに非空の `password` が記載されていた場合に警告ログを出し、`.env` での管理を促す。

### 1.1.1 src/core/watchlist.py — ウォッチリスト管理（複数リスト対応）

**責務**: 監視・取引対象銘柄（コード・会社名・セクター）を複数のウォッチリストとして
`watchlists.json` で永続化し、ダッシュボードGUIからの追加・削除・リスト切替を
プロセス再起動なしで反映する。取引は常に「アクティブな1リスト」に対して動作する。

**アクティブリストへの操作**（既存呼び出し元との互換API。常にアクティブリストに対して動作）:

| 関数 | 引数 | 戻り値 | 説明 |
|------|------|--------|------|
| `load(path, legacy_path)` | str, str | dict | 新形式JSONを読み込み。無ければ旧形式から自動移行 |
| `get_all()` | なし | list[dict] | `[{"code": ..., "name": ..., "sector": ...}, ...]` を返す |
| `get_codes()` | なし | list[str] | アクティブリストの銘柄コードのみのリストを返す |
| `get_all_codes()` | なし | list[str] | **全リスト**（アクティブ以外も含む）の銘柄コードを重複除去して返す。MLモデル学習データの母集団確保用（後述） |
| `get_names()` | なし | dict[str, str] | コード→会社名マッピングを返す |
| `get_sectors()` | なし | dict[str, str] | コード→セクターマッピングを返す（セクター集中チェック用） |
| `add(code, name)` | code, name: str | list[dict] | 追加（既存コードなら会社名を更新）し即座にファイル書き込み |
| `update_sector(code, sector)` | code, sector: str | None | セクターを更新（空文字なら何もしない） |
| `remove(code)` | code: str | list[dict] | 削除し即座にファイル書き込み |

**複数リスト管理**:

| 関数 | 説明 |
|------|------|
| `get_list_names()` | 全リスト名を返す |
| `get_active_list_name()` | アクティブなリスト名を返す |
| `create_list(name)` | 新規の空リストを作成し、アクティブに切替 |
| `delete_list(name)` | リストを削除（最後の1つは不可。アクティブなら自動的に他へ切替） |
| `switch_active(name)` | アクティブなリストを切替 |
| `rename_list(old, new)` | リスト名を変更（アクティブポインタも追従） |
| `export_list(name=None)` | 指定（省略時はアクティブ）リストを `{"name", "entries"}` で返す |
| `import_list(name, entries, overwrite)` | エクスポートされたデータからリストを作成・アクティブに切替 |

**入力検証（販売向けの堅牢化）**: 銘柄コードは `normalize_code()`（NFKC正規化で全角→半角）後、
`_validate_code()` で **ASCII英数（と `.`）・最大長 `MAX_CODE_LEN`(=12)** に限定して検証する
（不正なら `ValueError`）。`add()`（GUI単体追加）・`import_list()` の双方で適用する。
`import_list()` はさらに、信頼できないアップロードJSONを想定し `_validate_import_entries()` で
**リスト型であること・件数上限 `MAX_ENTRIES_PER_LIST`(=500)・各要素がオブジェクトであること**を
検証し、空コードのスキップ・重複除去・name/sector の長さ制限（各100文字）を行う。

`main.py` の各スケジュールジョブ（データ更新・損切りチェック・シグナルスキャン等）は
ループ開始時に `watchlist_store.get_codes()` を呼ぶため、GUIでの変更（銘柄の追加・削除や
アクティブリストの切替）は次回のジョブ実行（最短で5分後の損切りチェック）から反映される。
詳細は §4.2 を参照。

### 1.1.2 src/core/auth.py — ログイン認証情報・セッション管理

**責務**: ダッシュボードのユーザーID・パスワードの保存・照合と、ログインセッションの管理。

| 関数 | 引数 | 戻り値 | 説明 |
|------|------|--------|------|
| `load(path, session_ttl_hours=None)` | str, int? | なし | Authファイルのパス設定・存在すれば読み込み |
| `is_configured()` | なし | bool | 認証情報（ユーザーID・パスワードのハッシュ）が登録済みか |
| `create_user(username, password)` | str, str | なし | 初期設定。ユーザーID・パスワードをそれぞれ専用ソルトでハッシュ化して保存。既に設定済みなら `ValueError`。パスワード8文字未満も `ValueError` |
| `verify(username, password)` | str, str | bool | ユーザーID・パスワードの双方をハッシュ再計算＋`hmac.compare_digest`で照合 |
| `create_session()` | なし | str | セッショントークンを発行（`secrets.token_urlsafe(32)`） |
| `validate_session(token)` | str | bool | 有効か検証。期限切れは自動破棄 |
| `destroy_session(token)` | str | なし | セッションを破棄（ログアウト） |

**ユーザーID・パスワードの保存方式**: いずれも復号可能な「暗号化」ではなく、**それぞれ専用の
ソルトを持つ PBKDF2-HMAC-SHA256 ハッシュ**（16バイトランダムソルト・20万回反復）として保存する。
Authファイルが漏えいしても元のユーザーID・パスワードを復元できず、暗号鍵の管理も不要。
ファイルを見てもユーザーIDが判別できないため、`is_configured()` の判定や `verify()` の照合は
ハッシュの再計算＋`hmac.compare_digest`（定数時間比較）で行う。Python標準ライブラリ
（`hashlib`/`hmac`/`secrets`）のみで実装し、追加依存はない。

**保存先**: `data/auth.json`（既定。`dashboard.auth_file` で変更可）。
```json
{ "username_salt": "<hex>", "username_hash": "<hex>",
  "salt": "<hex>", "hash": "<hex>",
  "iterations": 200000, "algo": "pbkdf2_sha256", "created_at": "..." }
```
`.gitignore` 済みでgit管理されない。

**セッション**: メモリ内辞書（`token -> 期限`）で管理するため、**プロセス再起動で全セッションが
無効化**され、再ログインが必要になる（DBやファイルに永続化しない設計）。有効期限は既定168時間
（`dashboard.session_ttl_hours`）。

ダッシュボードからの利用方法は §1.12「ダッシュボード全体のアクセス認証」を参照。

### 1.2 src/core/scheduler.py — スケジューラ

**責務**: 定時処理の登録・実行管理（Asia/Tokyo固定）

**クラス: `TradingScheduler`**

| メソッド | 説明 |
|---------|------|
| `register(name, callback)` | コールバック関数を名前で登録 |
| `start()` | 全ジョブをAPSchedulerに登録して起動 |
| `stop()` | スケジューラを停止 |
| `is_market_open()` | 現在が東証取引時間内かチェック（静的） |
| `is_maintenance_window()` | 土曜深夜メンテ時間帯かチェック（静的） |

**取引時間判定ロジック**:
```
平日（月〜金）かつ
  前場: 9:00 <= 現在時刻 < 11:30
  または
  後場: 12:30 <= 現在時刻 <= 15:30
```

**登録ジョブ一覧**（すべて Asia/Tokyo）:

| ジョブID | 時刻 | 内容 |
|---------|------|------|
| `risk_reset` | 平日 8:25 | `RiskManager.reset_daily_counters()`（注文数・損失額リセット） |
| `token_refresh` | 毎日 8:30 | kabuステーションAPIトークン更新 |
| `morning_execution` | 平日 9:05 | 前日シグナルに基づく発注（ライブモードのみ） |
| `stop_loss_check` | 平日 9-15時 5分おき | 損切りライン到達チェック |
| `data_update` | 毎日 16:00 | yfinanceで当日分OHLCVを取得・DB反映 |
| `signal_scan` | 平日 16:20 | 翌営業日の売買候補スキャン（`data_update` より後ろの時刻に固定） |
| `db_backup` | 毎日 17:00 | DBバックアップ |
| `ml_retrain` | 日曜 2:00 | MLモデル週次再学習 |
| `reconcile_orders` | 15秒間隔（intervalトリガ） | `OrderManager.reconcile_open_orders()` による注文状態の定期照合。`TradingServices.reconcile_orders()` が `is_market_open()` でガードし市場時間外は何もしない |

> `data_update` と `ml_retrain` はいずれも `watchlist_store.get_all_codes()`（**全リスト**の
> 銘柄）を対象とする。取引対象は「アクティブな1リスト」（`get_codes()`）に限定されるが、
> データ取得とMLモデルの学習データはそれより広い母集団から確保し、リスト切替によって
> 学習サンプル数が急に減らないようにしている。

### 1.4 src/data/database.py — データベース

**責務**: SQLiteの初期化・テーブル定義・バックアップ

**接続設定**:
- WALモード有効（複数スレッドからの同時アクセス対応）
- `check_same_thread=False`（マルチスレッド許可）

**簡易マイグレーション（`_migrate_add_missing_columns`）**:
- `Base.metadata.create_all()` はテーブルの新規作成のみ行い、既存テーブルへの
  カラム追加は行わない。そのため `init()` 内で `PRAGMA table_info` を見て、
  モデル定義に対して不足しているカラムを `ALTER TABLE ADD COLUMN` で自動補完する
  （例: `backtest_runs.buy_threshold`/`sell_threshold` の追加）。データは失われない。

**`get_session()` — セッション取得（`@contextmanager`）**:
- `yield` でセッションを提供し、例外発生時は明示的に `rollback()`、`finally` で必ず `close()` を実行
- すべての呼び出し元は `with get_session() as session:` パターンで使用

> **注文ステータス体系**（`src/execution/order_status.py` に集約）
> - `PENDING`: 発注済み・約定/取消待ち
> - `PARTIALLY_FILLED`: 一部約定（残数は未約定）
> - `FILLED`: 全数約定
> - `CANCEL_REQUESTED`: 取消要求済み
> - `CANCELLED`: 取消成立
> - `CANCEL_FAILED`: 取消API失敗（注文が生存の可能性→要人手確認）
> - `REJECTED`: 証券会社に拒否された

>

### 1.7 src/strategy/labeling.py — トリプルバリア法ラベリング

**責務**: López de Prado のトリプルバリア法によるラベル生成とサンプル重み計算

| 関数 | 説明 |
|------|------|
| `get_daily_volatility(close, span)` | 日次リターンの指数加重標準偏差（動的ボラティリティ） |
| `triple_barrier_labels(df, pt_mult, sl_mult, max_holding, vol_span)` | 各行にラベルとイベント終了位置を付与 |
| `get_sample_weights(t_ends)` | ラベル重なりに基づくサンプル一意性重み |
| `build_training_set(df)` | 特徴量・ラベル・重みをまとめて生成（X, y, weights） |

#### トリプルバリアの3つの境界

```
エントリー価格 P、ボラティリティ v に対して
  上バリア（利益確定）  : P × (1 + tb_profit_mult × v)
  下バリア（損切り）    : P × (1 - tb_stop_mult × v)
  垂直バリア（時間切れ） : エントリーから tb_max_holding 日後

ラベル付け:
  上バリアに先に到達       → 1（利益が出る買い場）
  下バリアに先に到達       → 0（損切りになる）
  時間切れ                → 最終リターンの符号（>0で1、それ以外0）
```

#### サンプル一意性重み

```
各バーで同時進行中のイベント数（concurrency）を集計
イベントiの重み = そのイベント期間における 1/concurrency の平均
  → 多数のラベルが時間的に重なる期間のサンプルは情報重複が大きいため
    重みを下げ、IID仮定の崩れによる過学習を抑制する
```

### 1.8 src/strategy/ml_model.py — 機械学習モデル

**責務**: LightGBMモデルの学習・保存・推論

> **モデル選定根拠**: 2025-2026年の比較研究で、日足スイングのようにサンプル数が
> 限られる表形式金融データでは、勾配ブースティング（LightGBM）がTransformer系
> 深層学習（PatchTST/TFT/TimesNet等）より優位かつ計算効率・解釈性で勝るとされる。
> モデル本体はLightGBMを維持し、ラベリングをトリプルバリア法へ強化した。

#### `train(df, trigger, save)` — 学習フロー

`trigger` が指定された場合（`"weekly_schedule"` / `"manual"`）はCV結果・特徴量重要度を `model_metrics` テーブルに保存する。バックテスト内での学習時は `trigger=None` で呼び出してDBへの記録を抑制する。

`save=False` を指定するとモデルを `models/lgb_model.pkl` に書き込まない（バックテスト内での呼び出しで使用し、本番モデルの上書きを防ぐ）。デフォルトは `save=True`。



```
1. build_training_set() で特徴量・トリプルバリアラベル・サンプル重みを生成
2. データが100件未満 → ValueError
3. TimeSeriesSplit(n_splits=5) でCV
   - 各foldでLGBMClassifier学習（sample_weight付き）
   - early_stopping(20) で最適ステップ数を収集（best_n_list）
4. CV精度（平均・標準偏差）をログ出力
5. 各foldの最適ステップ数を平均して best_n_estimators を決定
6. 全データ＋サンプル重みで最終モデルを学習
7. models/lgb_model.pkl に保存
```

**LightGBMパラメータ**:

| パラメータ | 値 |
|-----------|-----|
| n_estimators | 200（early stoppingで調整） |
| learning_rate | 0.05 |
| num_leaves | 31 |
| random_state | 42 |

#### `predict_proba(model, df)` — 推論

```
入力: OHLCV DataFrame（またはすでに特徴量列を持つDataFrame）
  → FEATURE_COLS[0]（ma_cross_sm）が既に存在する場合はbuild_features()をスキップ
    （signal.py から build_features 済みのdfを渡す場合に二重計算を防ぐ）
  → 最新行のみ抽出
  → model.predict_proba() で上昇確率取得
出力: float（0.0〜1.0）
  ※ データ不足時は 0.5（中立）を返す
```

### 1.9 src/strategy/signal.py — シグナル生成

**責務**: ルールベース＋MLスコアを統合して売買シグナルを生成

**`generate()` の前処理**:
- `build_features(df)` を呼び出してテクニカル指標＋ML用特徴量を一括生成
- `compute_indicators()` ではなく `build_features()` を使うことで、`predict_proba()` への渡し時に二重計算を防ぐ

#### `compute_rule_score(df)` — ルールベーススコア計算

> **旧名 `_rule_score`**: バックテストエンジンから呼び出す必要があるため公開関数に昇格。



| 条件 | スコア変動 |
|------|----------|
| ゴールデンクロス（ma5がma25を上抜け） | +0.4 |
| デッドクロス（ma5がma25を下抜け） | -0.4 |
| RSI < 30（売られすぎ） | +0.3 |
| RSI > 70（買われすぎ） | -0.3 |
| 終値 < ボリンジャー下限 | +0.2 |
| 終値 > ボリンジャー上限 | -0.2 |
| MACDヒストグラム正転換 | +0.1 |
| MACDヒストグラム負転換 | -0.1 |

スコア範囲: -1.0 〜 +1.0（`max(-1.0, min(1.0, score))` でクリップ）

#### 統合スコア計算

```
MLスコア変換: (上昇確率 - 0.5) × 2   ← -1.0〜+1.0 に正規化

統合スコア = ルールスコア × 0.5 + MLスコア × 0.5

統合スコア >= +0.25 → BUY
統合スコア <= -0.25 → SELL
それ以外            → HOLD
```

> **閾値はスコアの実効レンジに合わせる**: ルールスコア（理論±1.0）とMLスコアを
> 0.5ずつ重み付き平均するため、統合スコアの実測レンジは約±0.3に収まる
> （実データ: 平均≈0、標準偏差≈0.11、最大≈0.32）。閾値 0.25 は平均+約2.2σで、
> 年2〜3往復程度のスイング頻度になる。理論上限1.0を前提とした旧値0.6は実効レンジ外で
> 一度も発火しなかった。バックテスト完了時の `[診断]` ログ（`_log_score_diagnostics`）で
> 候補閾値ごとのシグナル発生回数を確認し、銘柄・期間に応じて調整できる。

### 1.10 src/risk/manager.py — リスク管理

**責務**: 発注前のバリデーションと損切り判定

**クラス: `RiskManager`**

| メソッド | 説明 |
|---------|------|
| `can_place_order()` | 日次注文数上限・当日損失上限・未解決注文(UNKNOWN/CANCEL_FAILED)ガードの統合チェック。`(ok, reason)` を返す |
| `reset_daily_counters()` | 日次カウンタリセット（`_daily_order_count` と `_daily_loss_yen` 両方） |
| `restore_daily_state()` | 起動時にDBから当日（JST）約定済みTradeを集計し、日次カウンタを復元する |
| `record_loss(pnl)` | 損益を記録。負のpnlのみ `_daily_loss_yen` に累積する |
| `is_daily_loss_limit_reached()` | 当日損失上限チェック。`(over_limit, reason)` を返す |
| `increment_order_count()` | 注文カウンタをインクリメント |
| `calc_position_size(symbol, price, cash)` | 購入可能株数計算 |
| `check_max_positions(candidate_symbol)` | 最大保有銘柄数チェック（保有＋未約定BUY＋候補銘柄の集合で判定） |
| `check_sector_concentration(sector, candidate_notional)` | セクター集中率チェック（建玉の時価評価額ベース＋候補注文金額） |
| `should_stop_loss(symbol, price)` | 損切りライン判定 |
| `validate_buy(symbol, price, cash, sector)` | 買い注文の総合バリデーション |

#### 当日損失上限チェック

```
_daily_loss_yen: SELL 約定時に record_loss(pnl) で累積（損失のみ）
  ↓
can_place_order() で is_daily_loss_limit_reached() を呼び出し
  → _daily_loss_yen >= max_daily_loss → (False, "当日損失上限(N円)に達しました")
  → max_daily_loss == 0 → チェック無効（デフォルト動作）

毎営業日 8:25（scheduler.py の risk_reset ジョブ）に reset_daily_counters() を呼び、
_daily_loss_yen と _daily_order_count の両方をリセットする。
```

#### 日次カウンタの永続性（restore_daily_state）

`_daily_order_count`/`_daily_loss_yen` はプロセスのメモリ上のみで管理されるため、
当日損失上限に達して取引を停止した直後にプロセスが再起動すると、カウンタが
0にリセットされ安全装置が無効化されてしまう。これを防ぐため、`main.py` の起動時に
`risk.restore_daily_state()` を呼び、当日（JST）の `filled_at` を持つ Trade レコードから
注文数・損失額（`pnl < 0` の合計）を再集計してカウンタへ復元する。

#### validate_buy() の結線

`validate_buy(symbol, price, cash, sector)` は `signal_scan`（ペーパーモード）と
`morning_execution`（ライブモード）のBUYパスから呼び出され、`check_max_positions()` と
`check_sector_concentration(sector)` を発注前に必ず通過させる。セクターは
`watchlist_store.get_sectors()`（ウォッチリスト追加時に `market_data.lookup_sector()` で
yfinanceから自動取得・保存）から取得する。取得に失敗した場合は空文字となり、
セクター集中チェックはスキップされる（`check_max_positions()` は引き続き有効）。

呼び出し順は `can_place_order()` → `check_max_positions(candidate_symbol=symbol)` →
`calc_position_size()`（株数を確定）→ `check_sector_concentration(sector, candidate_notional=price*quantity)`。
**この注文自体の金額**をセクター集中度に加味してから判定するため、既存保有・未約定BUYの
引当だけでは弾けない「この注文を加えたら超過する」ケースも発注前に防げる
（外部レビュー指摘: 候補注文自体の金額が判定に入っていなかった）。

#### 未約定BUYを含む最大保有銘柄数チェック

`check_max_positions(candidate_symbol)` は保有Position銘柄数の単純カウントではなく、
「保有銘柄 ∪ 未約定BUYの銘柄 ∪ 候補銘柄」の**集合**で判定する
（外部レビュー指摘: 保有3/上限4のときに未約定BUYが2件あると、個別には通って
合計5銘柄になってしまう抜けがあった）。候補銘柄が既に保有/未約定であれば集合は
増えないため、既存ポジションへの追加買いは上限到達中でも通る。

#### 購入株数計算

```
上限金額 = 現金残高 × max_position_ratio（アクティブなリスクプロファイル由来。既定 low_risk では12%）
最大単元数 = floor(上限金額 / (株価 × 100))
購入株数 = 最大単元数 × 100
```

#### 損切り判定

```
損益率 = (現在値 - 平均取得単価) / 平均取得単価
損益率 <= stop_loss_pct（デフォルト-5%）→ 損切り発動
```

### 1.11 src/execution/order_manager.py — 発注管理

**責務**: 指値・成行注文の発注・ポジション更新・タイムアウト管理

**クラス: `OrderManager`**

**発注種別**:
- 通常のエントリ/エグジットは指値（`buy()` / `sell()`、`FrontOrderType=20`）
- 損切り・緊急全決済は成行（`sell_market()`、`FrontOrderType=10`・`Price=0`）。急変時に
  指値だと約定しない恐れがあるため、確実な約定を優先する

**`sell_market()` の `reason` とリスクゲートのバイパス**:
- `reason="normal"`（既定）: 新規発注と同じゲート（`can_place_order()`・`_has_pending_order()`）を適用
- `reason="stop_loss"` / `"emergency"`: 既存リスクを減らす退出操作のため、日次注文上限・
  当日損失上限・未解決注文ガードなどの**新規発注向けゲートをバイパス**する。
  代わりに `_cancel_open_orders_for_symbol()` で同銘柄の未約定注文を先にキャンセルしてから
  成行売りを送る（外部レビューで判明した重大な回帰: 緊急決済/損切りが新規発注と同じ
  ゲートに引っかかり、損失上限到達時にまさにポジションを閉じられない状態になっていた）
- `close_all_positions()` は `reason="emergency"`、損切り（`TradingServices.stop_loss_check()`）は
  `reason="stop_loss"` で呼ぶ

**スレッドセーフティ**:
- `_pending_orders` 辞書へのアクセスはすべて `self._orders_lock`（`threading.Lock`）で保護
- `_cancel_timeout_timer()` と `_timeout_cancel()` の両方がロックを取得してから辞書を操作

**日次注文上限・損失上限チェック**:
- `buy()` / `sell()` の先頭で `risk.can_place_order()` を呼び出し、注文数/損失上限のいずれかに達していれば即 `None` を返す
- `increment_order_count()` は「成功が確定した後」にのみ呼び出す（ペーパー: 即時成立する直前。
  ライブ: `Result==0` かつ `OrderId` 取得後）。API拒否や通信失敗で消費されないようにするため

**緊急決済・損切り（成行）**:
- `close_all_positions()` は各保有銘柄を `sell_market()`（成行）で決済する。成行のため板の
  現在値取得は不要（旧実装の指値＋0円指値ガードは廃止）
- `stop_loss_check()`（main側ジョブ）も損切り発動時は `sell_market()` を呼ぶ

**キャンセル失敗の扱い（`_timeout_cancel()`）**:
- `cancel_order()` が例外、または `Result != 0`（拒否）の場合は DB を `CANCELLED` にせず
  `CANCEL_FAILED` に更新し `alert()` で通知する（注文が証券会社側で生存している可能性があり、
  CANCELLED にすると実口座と乖離して二重発注・想定外建玉を招くため）
- 成功（`Result == 0`）時のみ `CANCELLED`

**SELL約定時の不整合検知**:
- `_update_position()` は SELL でポジションが存在しない場合、PnL集計・リスク反映をスキップし、
  状況を警告ログに出す（無関係な約定通知が来た場合などの不整合を見逃さないため）

**二重発注防止（ライブモードのみ）**:
- `buy()` / `sell()` でライブモード時に `_has_pending_order(symbol)` を呼び出す
- DBの trades テーブルで同銘柄の PENDING 注文が存在する場合はスキップ

**注文状態の収束 — WebSocketは補助トリガ、正本は `/orders` 照会**:
- 外部レビューで「WebSocket PUSHは本来板/価格情報主体であり、注文状態の正本として
  扱うのは不安全（イベント取り逃し・再起動・WS切断に弱い）」と指摘された。
  これを受け、状態確定は必ず `_status_from_api_order()`/`_extract_fill()` による
  `/orders` 照会結果で行う設計に統一した
- `on_order_event()`: 受信した `OrderID` のTradeが未約定（`OPEN_STATUSES`）なら
  `_reconcile_trade()`（その場で `get_orders()` を呼び該当注文を照会）に委譲するだけの
  薄いトリガ。既に確定済みのTradeは再照会しない
- `reconcile_open_orders()`: 未約定Tradeを一括で `get_orders()` 1回と突合し、
  `_sync_trade_with_order()` で収束させる。**定期実行**（スケジューラ `reconcile_orders`
  ジョブ、15秒間隔、`is_market_open()` 中のみ実働）し、WebSocketイベントの取り逃し・
  接続断・プロセス再起動を跨いでDB↔ブローカーのズレを検知・補正する
- `_sync_trade_with_order(trade, order)`: `_status_from_api_order()` で
  `OrderState/State==5` を**無条件で約定とは扱わない**。`cum_qty`（約定数量）を
  `None`/`0`/`>0` で明確に区別し、`cum_qty>=発注数量`→`FILLED`、`0<cum_qty<発注数量`→
  `PARTIALLY_FILLED`、`cum_qty==0`かつ終了状態→`CANCELLED` と判定する
  （外部レビューで判明した回帰バグ: 旧実装は `cum_qty=0`（取消/失効による0株約定終了）を
  Pythonの真偽値評価で偽と判定し発注数量にフォールバックしてしまい、**約定0株の注文を
  誤ってFILLEDにしていた**）。実約定単価・数量は `filled_price`/`filled_quantity` に記録し、
  増分のみポジションへ反映する。API照会で見つからない注文は `UNKNOWN`＋`alert()`

**起動時API同期 `sync_on_startup()`**:
- ライブモード起動時に kabuステーション API から現在の注文一覧を取得
- API に見つかった注文は実際の `State`/`CumQty` から DBステータスへ同期（5=終了かつ約定あり→FILLED 等）
- API に**全く見つからない**未解決注文は `CANCELLED` ではなく `UNKNOWN` とし `alert()` で通知する
  （約定済み注文を取消と誤判定しないため。`UNKNOWN` が残る間は新規発注を抑止）

#### ペーパーモードとliveモードの切り替え

```python
self._is_paper = config["trading"]["mode"] == "paper"
```

| モード | buy()/sell() の動作 |
|--------|-------------------|
| paper | DBに status=FILLED / filled_at=今 で即時記録。ポジションも即時更新。 |
| live | kabuステーションAPIで発注（`Result==0` かつ `OrderId` 非空を検証）→ WebSocketで約定確認後にポジション更新 |

**ライブモードでの発注結果検証**:
```
result = client.send_order(order)
result["Result"] != 0  → エラーログ・None 返却
result["OrderId"] が空  → エラーログ・None 返却
```

#### 注文IDの生成

| モード | 形式 |
|--------|------|
| paper | `PAPER-BUY-{symbol}-{uuid8桁}` |
| live | kabuステーションAPIが返す OrderId |

#### タイムアウトキャンセル

```
発注と同時に threading.Timer を設定
  → order_timeout_seconds（デフォルト300秒）後に cancel_order() を呼び出し
  → 約定確認イベントを受信した場合はタイマーをキャンセル
```

#### ポジション更新ロジック（BUY時）

```
既存ポジションあり:
  新平均取得単価 = (既存単価×既存株数 + 新単価×新株数) / (既存株数+新株数)
  株数 += 新株数

既存ポジションなし:
  新規Positionレコードを作成
```

#### ポジション更新ロジック（SELL時）

```
損益 = (売却価格 - 平均取得単価) × 売却株数
株数 -= 売却株数（最小0）
損益を対応するTradeレコードに記録（order_id で一意に特定）
```

> `order_id` を使ってTradeレコードを特定することで、同一銘柄の並行売却時に
> `symbol + side` による曖昧な照合が発生しないようにしている。

#### 統計ヘルパー関数

| 関数 | 説明 |
|------|------|
| `_compute_max_drawdown(cumulative_values)` | 累積損益リストから最大ドローダウン率を返す（負値、例: -0.08 = -8%） |
| `_compute_sharpe(daily_pnl)` | 日次損益リストから年率換算シャープレシオを返す（√252 でスケール） |

**最大ドローダウン計算**:
```
peak = 累積損益の現時点最大値
drawdown = (current - peak) / peak  ← peak > 0 の場合のみ
max_drawdown = min(全drawdown値)
```

**シャープレシオ計算**（無リスク金利=0として近似）:
```
sharpe = mean(daily_pnl) / std(daily_pnl) × √252
```

#### ダッシュボード全体のアクセス認証（ログイン + トークン）

販売（不特定多数への配布）に向けたセキュリティ強化として、ダッシュボード全体に
認証を導入している（`init_auth()` と `_auth_middleware`、認証情報は `src/core/auth.py`）。
ブラウザは**ログイン（ユーザーID・パスワード）**、curl等は**トークン**を併用する。

- **既定の待受は `0.0.0.0`**（LAN内の他端末からアクセス可・認証が自動で有効）。ローカル専用に限定する場合のみ `dashboard.host` を `127.0.0.1` にする。
- **認証が有効になる条件**: `dashboard.api_token`（または環境変数 `KABU_DASHBOARD_TOKEN`）が設定されている、または `host` が localhost 以外（LAN公開）。localhost かつトークン未設定なら認証なし（ローカル専用の後方互換）。
- **ログイン認証情報（Authファイル）**: `src/core/auth.py` が `data/auth.json`（`dashboard.auth_file`）に保存。
  パスワードは**復号可能な暗号化ではなく、ソルト付き PBKDF2-HMAC-SHA256 ハッシュ**（16バイトソルト・
  20万回反復、`hashlib`/`hmac`/`secrets` の標準ライブラリのみ）。一方向のため漏えいしても
  元パスワードを復元できず、鍵管理も不要。
- **初期設定（初回）**: Authファイルが無い状態で `GET /login` を開くと「初期設定」モードになり、
  `POST /api/setup {username, password}` でユーザーを作成する（作成は1度だけ。作成後はログイン状態になる）。
- **ログイン**: `POST /api/login {username, password}` で照合し、成功するとセッショントークンを
  発行して `kabu_session` Cookie（httponly, samesite=strict）に保存。セッションはメモリ管理で
  既定168時間（`dashboard.session_ttl_hours`）有効、プロセス再起動で無効化される。`POST /api/logout` で破棄。
- **認証OKの条件**: 有効な `kabu_session` セッション、または有効な `X-API-Token`/`?token=`/`kabu_token` トークン。
- **未認証時の挙動**: HTMLナビゲーション（`Accept: text/html` かつ `/api/` 以外）は `/login` へ 303 リダイレクト、
  API/XHR は 401。素通しパスは `/login`・`/api/login`・`/api/setup`・`/api/auth_status`。
- LAN公開でトークン未設定の場合は起動時に**`api_token` を自動生成**（curl用）し、認証を強制する。
- 認証有効時は `/docs`・`/openapi.json` も保護対象とし、未認証の第三者にAPI仕様を露出させない。
- 起動時にアクセスURLと（自動生成時の）トークンをコンソールへ一度だけ表示する（ログには出さない）。

#### 緊急決済エンドポイントの認証

- 上記のダッシュボード認証に加え、`X-Emergency-Token` ヘッダーが必須（FastAPI `Depends` で強制。二重の防御）
- トークンは起動時に `config.yaml` の `dashboard.emergency_token` から読み込む（空の場合はランダム生成）
- 自動生成したトークンは**起動時のコンソールに一度だけ表示**する（ログファイルには出力しない）
- フロントエンドの決済ボタン押下時に `prompt()` でトークン入力を求め、ヘッダーとして送信

### 1.13 src/backtest/engine.py — バックテストエンジン

**責務**: ウォークフォワード方式でのバックテスト実行とDB保存

#### `run_backtest(symbol, start, end, initial_capital, use_ml, buy_threshold=None, sell_threshold=None)` — 実行フロー

```
1. load_ohlcv(symbol, limit=2000) で全履歴を取得
2. build_features(full_df) で特徴量を一括計算（ルックアヘッドバイアスなし）
3. use_ml=True の場合: start以前のデータのみでMLモデルを学習（trigger=None）
4. 各テスト日（start〜end）についてループ:
     a. featured_df[date <= dt] でその日時点のデータにスライス
     b. ポジション保有中: 損切りライン（-5%）チェック
     c. シグナルスコア計算（compute_rule_score + predict_proba）
     d. BUYシグナル: 余力から購入株数を計算して買い
        SELLシグナル: 保有中なら決済
5. 最終日にポジション残存があれば END_OF_PERIOD で強制決済
6. BacktestRun（サマリー＋エクイティカーブ）をDBに保存
7. BacktestTradeRecord（全取引）をDBに保存
8. run_id を返す
```

**ルックアヘッドバイアス防止**:
- `build_features()` は全期間で一括計算しているが、各日のシグナル判定は `featured_df[featured_df.index <= dt]` でスライスした上で最新行のみ使用する
- ML学習時は `full_df[full_df.index.date < start]` のデータのみ使用

#### 閾値オーバーライド（探索的バックテスト用）

`buy_threshold`/`sell_threshold` を明示的に渡すと、アクティブなリスクプロファイル
（config の `strategy.buy_threshold`/`sell_threshold`）を**この実行だけ**上書きする。
config自体は書き換えないため、ライブ/ペーパー取引の設定には一切影響しない。

合成スコアの実効レンジは銘柄・期間によって大きく変わるため、閾値にわずかに届かず
取引数が0件になることがある（ダッシュボードの `[診断]` ログでスコア分布を確認できる）。
このオーバーライドにより、リスクプロファイルを切り替えずにその場で閾値を調整して
「どの閾値なら取引が発生するか」を探索できる。実際に使われた閾値は
`BacktestRun.buy_threshold`/`sell_threshold` に保存され、ダッシュボードの実行履歴・
詳細画面で確認できる。

#### 統計計算（バックテスト専用）

| 関数 | 説明 |
|------|------|
| `_max_drawdown(equity_curve, initial_capital)` | 資産曲線から最大ドローダウン率を計算 |
| `_sharpe_ratio(equity_curve)` | 資産曲線の日次リターンから年率シャープレシオを計算 |

| `places_real_orders(mode)` | 実発注しうるモード（live/semi_live）。二重確認・fail-closed・ポート競合中断の判定に使う |
| `reads_broker_api(mode)` | 実APIを読むモード（paper以外） |
| `uses_morning_execution(mode)` | 朝発注ジョブを実行するモード（paper以外） |

- **dry_run**: 実APIから板/余力は読むが `sendorder` しない。発注しようとした内容を
  `DRY_RUN` ステータス（OPEN/UNRESOLVED いずれにも非該当＝引当・建玉・未解決ガードに影響しない）
  の Trade として記録するだけ。
- **semi_live**: 通常の新規注文を承認キュー（`OrderApproval`）に積み、ダッシュボード承認で実発注。
  損切り・緊急決済（退出）は承認を介さず即時実発注する。

### 1.15.2 src/core/halt.py — 取引停止スイッチ（kill switch）

停止フラグを `data/trading_halt.json` に永続化する純粋な状態管理層（client/DBを知らない）。

| 関数 | 説明 |
|------|------|
| `load(path)` | 起動時に停止状態を復元（停止中なら critical ログ） |
| `is_halted()` | 新規発注を止めるべきか |
| `engage(reason)` / `release()` | 停止ON/OFF（永続化） |

- `RiskManager.can_place_order()` が `is_halted()` を見て新規発注を一律ブロックする。
  損切り・緊急決済は `sell_market(reason=...)` でこのゲート自体をバイパスするため停止中も実行可能。
- 副作用（未約定BUYキャンセル・成行決済）は `OrderManager.halt_trading()` が担う。
  解除 `resume_trading()` は未解決注文（UNKNOWN/CANCEL_FAILED）ゼロを条件とする。

### 1.15.3 src/core/preflight.py — ライブ起動前プリフライト

`paper` 以外の起動時に実行。実発注モードは致命的失敗で起動中断（fail-closed）、dry_runは記録のみ。

検査項目: API疎通（/orders・/positions・/wallet）／未解決注文ゼロ／ポート空き／

### 1.15.4 OrderManager の追加（モード分岐・承認・kill switch・状態）

## 1.18 注文モデル分離 OrderIntent / BrokerOrder / Fill（Phase 5 / 4.2）

### 1.18.1 背景

従来の `Trade`（1注文1行）は「意図（なぜ・何を）」「ブローカー注文（実際にkabuへ送った注文）」
「約定（個々のFill）」の3概念を兼ねており、(1) 部分約定明細がVWAPに潰れて再現性が無い、
(2) 1意図=複数注文（再発注・分割）を表せない、(3) PnLが平均単価会計でロット非対応、という
問題があった。**加算的な3層分離**（既存`trades`を温存し`order_intents`/`fills`を追加）で解消する。

```
OrderIntent (意図)  1 ──< BrokerOrder (=trades) ──< Fill (約定明細・FIFOロット台帳)
```

### 1.18.2 OrderIntent・Fillの生成（src/execution/order_manager.py）

- `_record_trade()`（Trade作成の唯一のchoke point）が、`intent_id` を渡されなければ
  新規 `OrderIntent` を生成して紐付ける。`intent_id` を渡された場合（semi_live承認実行時）は
  既存の意図を再利用する。`_initial_intent_status(trade_status)` が初期状態を決める
  （REJECTED→REJECTED、FILLED/DRY_RUN→COMPLETED、それ以外→SUBMITTED）。
- `buy()`/`sell()`/`sell_market()` は `source`（signal_scan/morning_execution/stop_loss/
  emergency/manual/broker_stop）を引数で受け取り `OrderIntent.source` に記録する。
  `services/trading.py` の各ジョブが該当する `source` を渡す。
- semi_live: `_enqueue_approval()` がこの時点で `OrderIntent(status="PENDING")` を作り
  `OrderApproval.intent_id` に紐付ける。`approve_order()` は同じ `intent_id` を
  `_live_buy`/`_live_sell`/`_live_sell_market` へ渡し、承認後のBrokerOrderが同じ意図へ
  紐づくようにする（承認待ち→承認→実発注の全行程が1つの意図で追跡できる）。

### 1.18.3 FIFOロット台帳（src/execution/lots.py）

平均単価会計からFIFOロット会計へ移行。session を受ける純粋な操作関数群（commitしない）。

| 関数 | 説明 |
|------|------|
| `record_buy_fill(session, broker_order_id, symbol, qty, price, filled_at, source)` | BUY約定を新規ロット（`Fill.remaining_qty=qty`）として記録 |
| `consume_fifo(session, broker_order_id, symbol, qty, sell_price, filled_at, source)` | SELL約定を `filled_at,id` 昇順の古いBUYロットから消費し、確定損益(`realized_pnl`)と消費数量を返す。保有不足時は消費できた分だけ処理（戻り値の`consumed`で呼び出し元が検知） |
| `rebuild_position(session, symbol, sector=None)` | 残存ロット（remaining_qty>0）の集計から `Position.quantity/avg_cost` を再構成。残ロット無しなら quantity=0（行は消さない） |

### 1.18.4 約定の適用 `_apply_fill()`（`_update_position()` の後継）

```python
_apply_fill(order_id, symbol, side, fill_qty, fill_price, filled_at, source, sector=None)
```

- BUY: `lots.record_buy_fill` でロット追加。SELL: `lots.consume_fifo` で確定損益を取得し
  `Trade.pnl += realized`、`RiskManager.record_loss(realized)`。
- 両side: `lots.rebuild_position` でPositionを再構成し、`_rollup_trade_fills()` が
  この注文(`broker_order_id`)に紐づく全Fillから `Trade.filled_quantity`/`filled_price`（VWAP）を
  再計算する派生列ロールアップを行う。
- 呼び出し元: paper（`buy`/`sell`/`sell_market`）は発注時点で即時呼ぶ。live/semi_liveは
  `_sync_trade_with_order()` の増分(delta_qty)反映として呼ぶ（`source="reconcile"`、
  on_order_event由来でも `reconcile_open_orders()` 由来でも同じ経路に集約）。

### 1.18.5 v2バックフィルマイグレーション（src/data/database.py）

`SCHEMA_VERSION=2`・`_MIGRATIONS[2] = _migrate_v2_order_model_backfill`。`engine.begin()` の
進行中Connectionに `Session(bind=conn)` を合流させ、`flush()` のみ行い commit は外側
（`_run_migrations`）に委ねる。

1. 全Trade（intent_id未設定のもの）にOrderIntentを生成して紐付け（`order_type`はSELLかつ
   price=0ならMARKET、`mode`は order_id の接頭辞 "PAPER-"/"DRYRUN-" から推定）。
2. 約定済み(FILLED/PARTIALLY_FILLED・filled_quantity>0)のTradeから、BUYを先に全件
   `lots.record_buy_fill` でロット化し、その後SELLを `filled_at,id` 昇順で
   `lots.consume_fifo` し、**`Trade.pnl` をFIFO再計算結果で上書き**する（旧・平均単価会計の
   値を置き換える。ユーザー確認済みの遡及再計算方針）。
3. Fillが生成された全symbolについて `lots.rebuild_position` でPositionを再構成。
4. 冪等性: `intent_id` 既設定のTrade・既にFillが存在するTrade（`broker_order_id`で判定）は
   スキップするため、複数回実行しても重複生成しない。

## 1.19 厳格レビュー対応（P0/P1。再レビュー 2026-06-20）

外部の厳格シニアレビュー（`kabu-auto_claude_fix_brief_strict_en.md`）の指摘のうち、
無監視ライブ運用に向けて必須（P0）と判断した4件と、運用上の高優先度（P1）2件を修正した。
ウォレット（現金残高）照合の自動化、リスクプロファイルのDB移行、アラート送信先の
抽象化、`OrderManager`の責務分割は、いずれも独立した大きめの作業になるため今回は
対象外とした（指摘自体は妥当だが、現金フロー台帳が無いなど前提となる基盤が別途必要、
または既存の安定動作への影響範囲が広いため、別途の実装サイクルで対応する）。

### 1.19.1 緊急全決済をブローカー基準に変更（P0-1）

`OrderManager.close_all_positions()` は、ライブ/semi_liveでは**ローカルDBの Position
ではなくブローカー `/positions`（`LeavesQty`）を正本として使う**。DBはWS取り逃し・
反映遅延・未知のバグ等で実口座とズレる可能性があり、緊急時にズレたDBを正本にすると
「実際は200株あるのに100株しか売らず残存」「実際は0株なのに不要な売りを送る」といった
事故になるため。`/positions` 取得自体に失敗した場合は実口座の状態を把握できないため、
DBに基づいた当てずっぽうの自動決済はせず critical alert を出して人手確認に委ねる
（fail-closed）。ペーパー/dry_runは実ブローカーが無いためDBを正本のまま使う。

### 1.19.2 退出注文の競合キャンセル失敗時はブロック（P0-4）

`_cancel_open_orders_for_symbol()` は全件キャンセル確定で `True`、1件でも失敗
（`CANCEL_FAILED`）があれば `False` を返すよう変更した。`sell_market()` の退出系
（`reason="stop_loss"/"emergency"`）はこの戻り値を見て、失敗していれば成行売りを
送信せず中断・alertする。旧実装はキャンセル結果を見ずに成行売りを送っており、
実口座に注文が残っているかもしれない状態へさらに新しい売り注文を重ねる危険があった。

### 1.19.3 終端部分約定の状態を分離（P0-2）

`order_status.py` に `PARTIALLY_FILLED_DONE` を追加した。`_status_from_api_order()`
は `State==5`（ブローカー側で確定終了）かつ `0<CumQty<発注数量` のとき、これまでの
`PARTIALLY_FILLED`（未約定＝`OPEN_STATUSES`に含まれる）ではなく
`PARTIALLY_FILLED_DONE`（`OPEN_STATUSES`に含まれない終端状態）を返す。

旧実装は「未約定のまま部分約定中」と「部分約定したまま取消/失効で確定終了」を
同じ `PARTIALLY_FILLED` で表現していたため、後者が `_has_pending_order()` /
`reconcile_open_orders()` の対象に残り続け、**その銘柄への新規発注が人手介入なしには
永久にブロックされる**バグがあった。`PARTIALLY_FILLED_DONE` はFILLEDと同様にタイマー
解除・`_apply_fill()`（残数分の約定反映）の対象にし、終端状態として正しく扱う。

### 1.19.4 ブローカー建玉ドリフト検知（P0-3・部分対応）

`OrderManager.reconcile_positions_with_broker()` を新設し、`TradingServices.
reconcile_orders()`（市場時間中、既存の `/orders` 照合と同じ定期ジョブ）から
`reconcile_open_orders()` と並行して呼ぶ。DBの `Position`（quantity>0）とブローカー
`/positions`（`LeavesQty`の銘柄別合計）を突き合わせ、以下のいずれかがあれば
ズレ（drift）として検知する: DBに建玉があるがブローカーに無い／ブローカーに建玉が
あるがDBに無い／数量が異なる。

ズレを検知したら critical alert を送り、既存の取引停止スイッチ（`halt.engage()`）を
作動させて新規発注を停止する（既に停止中なら理由を上書きしない）。kill switch は
損切り・緊急決済（`reason="stop_loss"/"emergency"`）をバイパスするため、退出操作で
建玉を減らす方向には影響しない。**ウォレット（現金残高）照合は対象外**（発注/約定/
入出金を追う現金フロー台帳が無く「期待される現金残高」を算出できないため。台帳の
新設は別の実装サイクルで検討する）。

### 1.19.5 多重起動防止のプロセスロック（P1-1）

`src/core/process_lock.py` を新設。`data/kabu_auto.lock` にPIDを書き込み、既存ロックが
あれば対象PIDのプロセスが実際に生きているか確認する（Windowsは`OpenProcess`、POSIXは
`os.kill(pid, 0)`）。生きていれば多重起動と判定（前回が異常終了した場合のstale lockは
上書きして起動を許可）。`main.py` はモード判定直後に取得を試み、paper以外
（dry_run/semi_live/live）は競合時に `sys.exit(1)` で起動中断、paperは警告のみで続行する。
正常シャットダウン時（`KeyboardInterrupt`）に `release()` で解放する。

既存のダッシュボードポート競合チェックだけでは、ポートが衝突しない設定や検知前の
タイミング次第で多重起動を見逃す可能性があったため、プロセスレベルで直接検知する
独立した安全網として追加した。

### 1.19.6 dry_runの本番口座データ読み取り警告（P1-3）

`src/core/preflight.py` に `_check_dry_run_production_data()` を追加。`mode=="dry_run"`
かつエンドポイントが検証用ポート(18081)以外（=本番用）の場合、warningレベルの
チェックとして可視化する。dry_runは`sendorder`を呼ばないため致命的ではないが、
検証目的で起動したつもりが本番口座の実際の余力・建玉を読み取っていることに
気づかないまま運用するリスクを防ぐ（起動は止めない）。

`main.py` の運用状態バナー直後でも同条件で WARNING を出し、運用者の誤認を二重に防ぐ。

### 1.19.7 プロセスロックの解放を atexit に一元化（フォローレビュー対応）

`process_lock.release()` を `main.py` の `atexit.register()` に登録する（ロック取得
成功時のみ）。これにより `KeyboardInterrupt` だけでなく、起動途中の `sys.exit(1)`・
予期せぬ例外を含むあらゆる正常終了経路でロックファイルが解放される。`release()` は
自プロセスが書いたロックのときのみ削除するため、多重呼び出しや他プロセスのロックに
対して安全（stale lock は次回起動時に上書き検知もできるが、正常終了時に残さない方が
望ましいため一元化した）。

### 1.19.8 paper多重起動の config 制御（フォローレビュー対応）

`config.yaml` の `runtime.allow_multiple_paper_instances`（既定 `false`）で paper の
多重起動許可を切り替える。paper は実資金リスクは無いが、DB・ログ・バックテスト結果・
ダッシュボード状態の汚染リスクがあるため、既定では paper でも多重起動を中断する
（テストデータ整合性を優先した安全側の既定）。意図的に複数の paper インスタンスを
動かしたい場合のみ `true` にする。paper 以外（dry_run/semi_live/live）は設定に
かかわらず常に中断する。

## 1.20 意図的に見送った決定事項（Deferred Decisions / ADR）

将来の保守者・AIエージェントが「なぜ実装されていないのか」を誤解し、意図せず
“修正”してしまうことを防ぐため、外部レビューで指摘されたが本サイクルで意図的に
見送った項目とその理由を明示する。各項目は妥当性を認めた上で、前提となる基盤の
不足または既存の安定動作への影響範囲の広さから、別サイクルでの対応とした。

| 見送った項目 | 理由 | 対応の前提・順序 |
|---|---|---|
| ウォレット（現金残高）照合の自動化 | 現金フロー台帳が無く「期待される現金残高」を算出できない。手数料・税・受渡タイミング・拘束金・ブローカー側手動操作・未反映約定などで正当な差異が生じ、台帳なしの自動照合は誤検知・誤補正を招く | ①Fill台帳 → ②手数料/税/受渡/拘束金の記録 → ③現金台帳 → ④ウォレットdrift検知 → ⑤自動補正はせず警告のみ |
| リスクプロファイルの JSON→DB 移行 | UI・履歴・import/export・マイグレーションへ影響が広い。現行のJSON実装が安定しているため別サイクルで扱う | 既存JSONの安定運用を確認の上、DBスキーマ設計＋移行を独立サイクルで |
| アラート送信先の抽象化（LINE依存の解消） | LINE Notify 依存は将来リスクだが、現行は安定。当面の最小対応として送信失敗をログ＋ヘルス状態に出すに留める | プロバイダ抽象（Discord/Slack/email/webhook）＋送信失敗の `/api/health` 露出 |
| `OrderManager` の責務分割 | 最重要のライブ発注経路に回帰バグを入れるリスクが高い。リファクタ前にテストを増やすのが正しい順序 | 経路ごとのテスト拡充 → Gateway/Repository/Reconciler/FillService等へ段階分割 |
| ダッシュボード既定 host を `0.0.0.0` のまま維持 | プロダクト判断。LAN公開時はログイン認証の強制・Cookieセッション・`X-API-Token`・平文HTTP警告を備え、内部的に一貫している | ローカル専用にしたい場合は `127.0.0.1` に変更可能（README参照） |

> これらは「未実装の欠陥」ではなく「意図的な先送り」である。再実装を検討する場合は
> 上表の前提・順序を満たしてから着手すること。

### 2.3 ペーパーモードの仮想ウォレット

ペーパーモードの発注サイズ算定（`risk.calc_position_size()` の `cash` 引数）には、
固定額（旧実装）ではなく以下で算出する仮想ウォレット残高を使う:

```
利用可能資金 = paper_initial_capital（config.trading、既定50万円）
              + 累積実現損益（Trade.pnl の合計）
              − 現在の建玉簿価（保有中Positionの avg_cost × quantity の合計）
```

固定額のままだと複数銘柄に対して資金制約が効かず、実現損益も反映されないため、
実際の口座挙動に近づけている（`src/services/trading.py::_paper_available_cash()`）。

## 3. エラー処理方針

| 状況 | 対応 |
|------|------|
| kabuステーション接続失敗（起動時・ペーパー） | ログ警告を出してペーパーモードで継続 |
| kabuステーション接続失敗（起動時・**ライブ**） | fail-closed: `logger.critical` を出し `sys.exit(1)` で起動を中断する（口座状態を把握できないまま発注ロジックが動く危険を避ける） |
| ライブモード環境変数未設定 | `CONFIRM_LIVE_TRADING=true` がなければ起動時に `sys.exit(1)` |
| ダッシュボードのポート競合 | `is_port_available()` で起動前に確認。ライブモードでは `sys.exit(1)`、ペーパーは警告ログのみで継続 |
| 起動時PENDING注文不明 | `sync_on_startup()` でAPIと照合。見つかれば実状態へ同期、API未存在は `UNKNOWN` に（CANCELLED にはしない）＋`alert()` |
| 未約定注文の定期照合 | `reconcile_open_orders()`（15秒間隔・市場時間中）でDBをブローカーの実状態へ収束。`OrderState==5` を無条件でFILLED扱いしない |
| キャンセルAPI失敗 | DBを `CANCEL_FAILED` に（CANCELLED にしない）＋`alert()`。未解決の間は新規発注を抑止 |
| 緊急決済・損切りが新規発注ゲートで止まる | `sell_market(reason="emergency"/"stop_loss")` は日次上限・損失上限・未解決注文ガードをバイパスし、競合する未約定注文を先にキャンセルしてから退出する |
| 未約定中の多重発注 | `RiskManager` が未約定BUYの引当を余力・セクター集中・最大保有銘柄数（銘柄集合）から控除 |
| APIトークン更新失敗 | LINE Notifyでアラート送信 |
| WebSocket切断 | 自動再接続（指数バックオフ、最大60秒） |
| 注文発注失敗（Result≠0 / OrderId欠落） | エラーログを記録して None を返す、処理スキップ |
| 二重発注 | _has_pending_order() でDBチェック、PENDING 存在時はスキップ（退出系は例外） |
| WebSocket約定通知の重複 | on_order_event() は未約定Tradeのみ再照会するため、確定済みは再処理しない |
| 当日損失上限到達 | can_place_order() が False を返し新規発注を停止 |
| データ取得失敗（yfinance） | ログ警告、当該銘柄をスキップ |
| ML推論失敗 | ログ警告、MLスコアを0.0（中立）として処理を継続 |
| DB操作エラー | SQLAlchemyの例外をcatch、ログ記録 |

### 4.2 watchlists.json（複数ウォッチリスト、GUIから編集可）

複数のウォッチリスト（例: "メイン" "高配当株" "グロース株"）を保持し、その中の
**1つだけがアクティブ**になる。取引（signal_scan/morning_execution/stop_loss_check 等）は
常にアクティブなリストに対して動作する。

```json
{
  "active": "メイン",
  "lists": {
    "メイン": [
      { "code": "8306", "name": "三菱UFJフィナンシャル・グループ", "sector": "Financial Services" },
      { "code": "7203", "name": "トヨタ自動車", "sector": "Consumer Cyclical" }
    ],
    "高配当株": []
  }
}
```

`src/core/watchlist.py` が読み書きを担当する。すべての既存API（`get_codes()`/`get_names()`/
`get_sectors()`/`add()`/`remove()`/`update_sector()`）は常にアクティブリストに対して
透過的に動作するため、`main.py` 側のジョブ実装は変更不要。

**旧形式からの自動移行**: 旧形式（フラットなリストのみ）の `watchlist.json` が存在し、
新形式の `watchlists.json` が存在しない場合、`load()` は自動的に旧データを
`{"active": "メイン", "lists": {"メイン": <旧データ>}}` へ変換して `watchlists.json` に保存する
（旧ファイルは削除しない。`.gitignore` で追跡対象外）。

**複数リスト管理API**:

| エンドポイント | 説明 |
|---|---|
| `GET /api/watchlists` | 全リスト名とアクティブなリスト名を返す |
| `POST /api/watchlists` `{name}` | 新規の空リストを作成し、アクティブに切替 |
| `DELETE /api/watchlists/{name}` | リストを削除（最後の1つは不可。アクティブなら自動的に他へ切替） |
| `POST /api/watchlists/active` `{name}` | アクティブなリストを切替（次回のジョブ実行から反映） |
| `PUT /api/watchlists/{name}` `{new_name}` | リスト名を変更 |

**外部出力（エクスポート）・取込（インポート）**:

| エンドポイント | 説明 |
|---|---|
| `GET /api/watchlist/export?name=<省略可>` | 指定リスト（省略時はアクティブ）をJSONファイルとしてダウンロード |
| `POST /api/watchlist/import` `{name, entries, overwrite}` | エクスポートされたJSONの `entries` から新規（または上書き）リストを作成し、アクティブに切替 |

ダウンロードファイル名に日本語等の非ASCII文字を含むリスト名を使う場合、HTTPヘッダーは
Latin-1必須のため `Content-Disposition` は RFC 6266 の `filename*=UTF-8''<percent-encoded>`
形式で渡す（ASCII の `filename=` はフォールバック用）。

### 4.3 リスクプロファイル切替（risk_profile.json）

**目的**: 日次の損益を見ながら「ハイリスク・ハイリターン」⇔「ローリスク・ローリターン」の
2択をダッシュボードから切り替えられるようにする。

**仕組み**（`src/core/risk_profile.py`）:
- `config.yaml` の `risk_profiles` に `low_risk`（ローリスク・ローリターン）と
  `high_risk`（ハイリスク・ハイリターン）の2プロファイルを定義
- アクティブなプロファイル名は `risk_profile.json`（`{"active": "low_risk"}`）に永続化。
  既定は `low_risk`（live運用へ進んだ場合の安全側デフォルト）
- 切替時、プロファイルの各値を `config` の trading / strategy セクション辞書へ**インプレース反映**する。
  `RiskManager` は trading 辞書の参照を保持し、strategy 側は `generate()` ごとに読み直すため、
  発注サイズ（`max_position_ratio`）・損切り幅（`stop_loss_pct`）・最大保有数（`max_positions`）・
  セクター集中（`max_sector_ratio`）・当日損失上限（`max_daily_loss`）・売買閾値
  （`buy_threshold`/`sell_threshold`）に即座に反映される
- 起動時 `main.py` が `risk_profile_store.load()` を呼び、永続化された（無ければ `active_risk_profile`）
  プロファイルを適用する

**API**: `GET /api/risk_profile`（現在値と全プロファイル）/ `POST /api/risk_profile {name}`（切替）。
`GET /api/status` にも `risk_profile`（アクティブ名）を含める。

**設計方針**: 単に1パラメータを動かすのではなく、ポジションサイズ・損切り幅・分散度・
シグナル選別の厳しさ・損失許容額の5軸を「リスクを取る方向」に一貫して連動させている
（ローリスク側は小さく張って早めに損切り・広く分散・強いシグナルのみ採用、ハイリスク側は
大きく集中投資して損切りを広げ取引頻度も上げる）。

| プロファイル | 性格 | max_position_ratio | stop_loss_pct | max_positions | max_sector_ratio | max_daily_loss | 売買閾値 |
|---|---|---|---|---|---|---|---|
| `low_risk` | ローリスク・ローリターン（既定） | 12% | -4% | 6 | 30% | ¥15,000 | ±0.14 |
| `high_risk` | ハイリスク・ハイリターン | 35% | -10% | 4 | 55% | ¥60,000 | ±0.08 |

> **売買閾値の根拠**: `combined_score` はイベント駆動のルールスコア（MAクロス/RSI極値/
> BBブレイク/MACDクロス）とMLモデルの予測確率を合成したものであり、何も起きない日は
> ルールスコア=0になるため、実際の値域はMLモデルの精度に強く依存する。当初の閾値
> （low_risk: ±0.32 / high_risk: ±0.18）は、実運用での実測スコア分布（2026-06時点:
> 平均0.043・標準偏差0.043・観測最大0.074）を大きく上回っており、シグナルが一度も
> 発生しない状態になっていた。これを受けて実測分布に基づき再調整した
> （high_risk ≈ 平均+0.8σ、low_risk ≈ 平均+2.3σ）。MLモデルの精度向上やルールベース
> 条件の見直しでスコア値域が変われば、この閾値も合わせて見直す必要がある。

### 4.4 日時のタイムゾーン方針

DBに保存する全ての `DateTime`（`Trade.filled_at`・`Position.opened_at/updated_at`・
`Signal.generated_at`・`ModelMetrics.trained_at` など）は **JST naive** で統一する。
スケジューラ・発注・ML はいずれも JST であり、`morning_execution` の cutoff 比較と
`Signal.generated_at` の基準を一致させないと、前日シグナルを9時間ぶん取りこぼして
ライブ自動売買が成立しない。ダッシュボードの当日損益判定（`enhanced_summary`）も同じ
JST naive を用い、フロントは JST naive をそのままローカル時刻として解釈する
（`generated_at` に末尾 `Z` を付けない）。

現在時刻の取得は `src/core/clock.py` の `now()` / `today()` に一元化する。`now()` は
`zoneinfo.ZoneInfo("Asia/Tokyo")` で明示的にJSTを求めたうえで tzinfo を外した naive を返すため、
ホスト機がJST以外でも一貫してJSTで動作する（`zoneinfo` が使えない環境ではローカル時刻の
naive にフォールバック。Windows向けに `tzdata` を依存に追加）。UTC保存への全面移行は上記
cutoff 比較を壊すため行わない。アプリ全体で `datetime.now()` 直書きは避け `clock.now()` を使う。
