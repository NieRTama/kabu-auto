### バージョン採番ルール

`メジャー.中規模.小規模.軽微` の4区分で管理する。

| 区分 | 位置 | 規則 | 例 |
|------|------|------|----|
| メインリリース | 1桁目 | 製品としての大規模リリース。当面 `1` 固定 | **1**.x.x.x |
| 中規模な改修 | 2桁目 | 新サブシステム・主要機能の追加（バックテスト・認証等）ごとに +1 | 1.**7**.x.x |
| 小規模な改修 | 3桁目 | 既存機能の拡張・補助機能の追加ごとに +1 | 1.7.**5**.x |
| 軽微な改修 | 4桁目 | バグ修正・ドキュメント整合・CI修正・設定微調整ごとに a→b→c…と更新 | 1.7.5.**c** |

> 軽微（4桁目）は上位区分（中規模・小規模）が上がるたびに `a` にリセットする。
> 例：メインリリース1回・中規模7回・小規模5回・直近の小規模以降に軽微3回 → `1.7.5.c`。

## 1. システム概要

### 1.1 目的

日本株（東京証券取引所）を対象としたスイングトレード（数日〜数週間保有）の完全自動売買システム。auカブコム証券のkabuステーションAPIを使用し、ルールベース＋機械学習ハイブリッド戦略でシグナルを生成・発注する。

### 1.2 対象ユーザー

個人投資家（システム管理者兼利用者）

### 1.3 運用形態

| 項目 | 内容 |
|------|------|
| 稼働形態 | 24時間365日常駐起動 |
| 取引時間 | 東証前場 9:00〜11:30 / 後場 12:30〜15:30（平日のみ） |
| 取引モード | paper（ペーパートレード）/ live（本番） |
| 運用資金 | 50万円以下（小規模） |

### 2.2 ディレクトリ構成

```
kabu-auto/
├── main.py                  # エントリポイント
├── config.yaml              # 設定ファイル（機密情報・銘柄リストは含まない）
├── watchlists.json          # ウォッチリスト（複数リスト対応・銘柄コード・会社名・セクター、GUIから編集可）
├── risk_profile.json        # アクティブなリスクプロファイル（GUIから切替・git管理外）
├── .env                     # 機密情報（kabuステーションのAPIパスワード等、git管理外）
├── .env.example             # .env のテンプレート
├── requirements.txt         # 依存パッケージ
├── src/
│   ├── api/                 # kabuステーションAPIクライアント
│   ├── core/                # 設定・ウォッチリスト・ログ・スケジューラ・アラート
│   ├── data/                # DB管理・市場データ取得
│   ├── strategy/            # テクニカル指標・MLモデル・シグナル生成
│   ├── execution/           # 発注・ポジション管理
│   ├── risk/                # リスク管理
│   └── dashboard/           # Webダッシュボード（FastAPI）
├── frontend/                # ダッシュボードUI（HTML/JS）
├── models/                  # 学習済みMLモデル（pickle）
├── data/                    # SQLite DB・ログ・バックアップ
│   └── backups/
└── docs/                    # 設計書
```

> **設定ファイルの分離方針**: `config.yaml` はgit管理され複数環境で共有される静的設定、
> `watchlists.json` はダッシュボードGUIから動的に追加・削除される銘柄リスト（複数リスト対応）、
> `risk_profile.json` はGUIから切り替えるアクティブなリスクプロファイル（個人用・git管理外）、
> `.env` は `.gitignore` 済みで各環境のみに存在する機密情報（APIパスワード等）。
> これらを分離することで、`git pull` 時に個人設定（パスワード・ウォッチリスト・リスク設定）が
> リポジトリ側の更新と競合する問題を避けられる。

## 6. 取引戦略概要

### 6.1 ハイブリッドシグナル

```
統合スコア = ルールベーススコア × 0.5 + MLスコア × 0.5  （理論レンジ -1.0 〜 +1.0）

統合スコア >= +0.25 → BUY シグナル
統合スコア <= -0.25 → SELL シグナル
それ以外            → HOLD
```

> **閾値の根拠**: 上の ±0.25 は `config.yaml` の `strategy` セクションの基準値だが、
> 実運用ではアクティブなリスクプロファイル（既定 `low_risk`: ±0.14 / `high_risk`: ±0.08）の
> 値で上書きされる。再調整の経緯は[詳細設計書](詳細設計書.md)を参照。
> ルール／MLスコアを重み付き平均するため統合スコアの実効レンジは狭く、実測スコア分布
> （2026-06時点: 平均0.043・標準偏差0.043）に合わせて閾値を引き下げている。
> 理論上限 1.0 を前提とした旧値は実効レンジ外で発火せず、要調整パラメータである。

### 6.2 使用テクニカル指標

| 指標 | パラメータ | 用途 |
|------|-----------|------|
| 移動平均線 | 短期5 / 中期25 / 長期75 | MAクロスでトレンド判定 |
| RSI | 14日 | 売られすぎ(30以下)/買われすぎ(70以上) |
| ボリンジャーバンド | 20日 / 2σ | バンド外れで反転シグナル |
| MACD | 標準設定 | ヒストグラム転換でシグナル |

> 全指標は `pandas` のみで実装（`pandas-ta` 非依存）。
> `pandas-ta==0.3.14b0` が Python 3.11 向けに PyPI で配布されていないため、
> SMA/RSI/BBands/MACD を `src/strategy/indicators.py` 内で直接計算する。

### 6.3 機械学習モデル

| 項目 | 内容 |
|------|------|
| アルゴリズム | LightGBM（二値分類） |
| 予測対象 | 損切り前に利益目標へ到達するか（トリプルバリア法 1/0） |
| 特徴量 | テクニカル指標10種 |
| ラベリング | トリプルバリア法（López de Prado）＋サンプル一意性重み |
| 学習方式 | 時系列交差検証（5分割）→全データで最終学習 |
| 再学習 | 毎週日曜2:00 |

> 2025-2026年の比較研究では、日足スイングのようなサンプル数が限られる表形式
> 金融データでは勾配ブースティング（LightGBM）がTransformer系深層学習より
> 優位とされる。最新トレンドを踏まえモデルはLightGBMを維持し、ラベリングを
> 最新ベストプラクティスのトリプルバリア法へ強化している。

## 7. リスク管理概要

> 数値はアクティブなリスクプロファイルで決まる（既定 `low_risk`。下表は代表値）。
> プロファイル別の正確な値は[詳細設計書](詳細設計書.md)・`config.yaml` を参照。

| リスク | 対策 |
|--------|------|
| 過大投資 | 1銘柄あたり最大投資比率を制限（low_risk:12% / high_risk:35%） |
| 急落 | 損切りライン（low_risk:-4% / high_risk:-10%）で強制成行売却 |
| 過集中 | 同一セクター集中上限（low_risk:30% / high_risk:55%） |
| 流動性不足 | 指値注文＋5分タイムアウトキャンセル |
| 過剰発注 | 1日の注文数上限100件 |
| 連続損失 | 当日損失上限（low_risk:¥15,000 / high_risk:¥60,000）を超えたら発注停止（再起動後もDBから復元） |
| 二重発注 | DB未約定チェック＋15秒間隔の状態照合により同銘柄への重複発注を防止 |
| ギャップ | 最大同時保有銘柄数を制限（low_risk:6 / high_risk:4）してリスク分散 |
| DB損失 | 日次バックアップ（30日分） |
| システム異常 | ダッシュボードから緊急全決済ボタン（リスクゲートをバイパスして確実に決済） |
| 誤起動 | ライブモード起動時にCONFIRM_LIVE_TRADING=true 環境変数を必須化 |
| 状態誤認 | ライブ起動時のトークン取得失敗・ポート競合は起動中断（fail-closed） |

## 9. 制約・前提条件

- kabuステーション（Windowsアプリ）が常時起動していること
- PCのスリープを無効化していること（スリープでAPI切断）
- kabuステーションのAPIトークンは毎朝8:30に自動更新
